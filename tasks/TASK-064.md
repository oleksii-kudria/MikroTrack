# TASK-064 - Fix offline -> online session timer reset after reconnect

## Контекст

Після виконання TASK-063 проблема з таймером для пристрою залишилась частково невирішеною.

Поточна поведінка:
- пристрій коректно переходить `online -> idle -> offline`
- для `offline` коректно оновлюється `offline_since`
- після повторного підключення до мережі пристрій переходить у `online`
- але `online_since` НЕ скидається на новий час
- таймер `elapsed_seconds` / `presence_duration_seconds` продовжує відлік зі старої, вже завершеної сесії

Очікувана поведінка:
- перехід `offline -> online` повинен завжди створювати нову online-сесію
- при такому переході `online_since` має встановлюватися в поточний момент
- `idle_since` має очищатися
- `offline_since` має очищатися
- таймер у web UI та API повинен починатися з `00:00`

---

## Виявлена причина проблеми

Проблема знаходиться в `app/persistence.py`, у логіці `_apply_stable_timestamps()`.

Зараз код недостатньо жорстко враховує вже зафіксований стан `offline` з попереднього snapshot.

Поточна помилка:
- якщо в попередньому snapshot пристрій уже мав `offline_since`
- при наступному циклі система знову намагається вивести effective state через `_derive_device_state()` та `_idle_timeout_exceeded()`
- через це попередній стан може бути оброблений не як завершена offline-сесія, а як продовження старої online/idle-сесії
- у результаті reconnect не завжди проходить як повноцінний `offline -> online`
- `online_since` успадковується зі старої сесії, хоча повинен створюватися заново

---

## Що потрібно зробити

### 1. Виправити визначення previous effective state
У `app/persistence.py` в `_apply_stable_timestamps()` потрібно змінити логіку так, щоб:

- якщо в попередньому snapshot вже існує валідний `offline_since`
- попередній effective state безумовно вважався `offline`
- у такому випадку не треба повторно визначати його через timeout або інші непрямі ознаки

Очікуване правило:
- `previous.offline_since != null` -> `previous_effective_state = "offline"`

### 2. Гарантувати створення нової сесії при reconnect
Перевірити і, за потреби, посилити логіку переходу:

```text
offline -> online
```

Щоб при такому переході завжди виконувалось:
- `online_since = now`
- `idle_since = null`
- `offline_since = null`
- `state_changed_at = now`

### 3. Не допустити продовження старої online-сесії
Потрібно виключити сценарій, коли:
- попередня сесія вже завершилась переходом в `offline`
- але нове підключення використовує старий `online_since`

Стара сесія повинна вважатися завершеною остаточно.

### 4. Перевірити узгодженість persistence та API
Перевірити, що після виправлення:
- snapshot містить правильні `online_since / idle_since / offline_since`
- API не повертає застарілий `online_since`
- `presence_duration_seconds` і `elapsed_seconds` рахуються від нової сесії

## Критерії приймання

Задача вважається виконаною, якщо:

- `offline_since` у попередньому snapshot коректно примушує попередній effective state до `offline`
- reconnect після `offline` завжди створює нову online-сесію
- `online_since` більше не переноситься зі старої завершеної сесії
- `idle_since` та `offline_since` очищаються коректно
- API повертає правильний timer після reconnect
- тести покривають regression-сценарій
- за потреби оновлені журнали подій
- за потреби оновлена документація українською та англійською

## Очікуваний результат

Після виправлення система повинна стабільно працювати так:

```text
online -> idle -> offline -> online
```

де:
- `offline` завершує попередню сесію
- наступний `online` починає нову сесію
- таймер online починається заново з `00:00`
- web UI та API показують новий session timer без успадкування значення з минулої сесії
