# TASK-084 - Align documentation, deployment examples, and runtime behavior before Phase 2

## Опис

Перед переходом до Phase 2 потрібно привести документацію MikroTrack до повністю узгодженого стану з поточною реалізацією.

Зараз репозиторій уже в робочому стані, але в документації ще залишаються місця, які можуть вводити в оману:

- Quick Start містить заглушки або занадто загальні приклади
- частина описів не повністю відображає актуальну поведінку collector / API / persistence / web UI
- приклади volume mapping та deployment path потрібно уніфікувати
- потрібно чітко розділити expected behavior, warnings, troubleshooting та real errors
- потрібно підготувати документацію так, щоб перед Phase 2 вона була стабільною опорною точкою для подальших задач

Ця задача не додає новий функціонал. Її мета - зробити документацію, приклади запуску та опис runtime behavior повністю узгодженими з поточним станом репозиторію.

---

## Що потрібно зробити

### 1. Перевірити README.md на повну узгодженість з поточним кодом

Потрібно перевірити і оновити:

- Quick Start
- опис architecture
- перелік collected data sources
- опис persistence
- опис event-driven diff
- опис web UI behavior
- список documentation links

Виправити все, що:
- застаріло
- не відповідає поточній реалізації
- звучить занадто загально
- може збити з пантелику нового користувача або розробника

---

### 2. Уніфікувати deployment приклади

У документації не повинно бути суперечливих прикладів на тему:

- `PERSISTENCE_PATH`
- Docker volume mapping
- host path vs relative path
- container path
- спосіб запуску через `docker compose`

Потрібно привести приклади до одного узгодженого варіанту або чітко пояснити, які варіанти допустимі.

---

### 3. Уточнити фактичні джерела даних collector

Документація має чітко описувати, що зараз реально використовує collector:

- DHCP leases
- ARP table
- bridge host
- interface MAC data
- optional / unsupported wireless interface handling

Якщо певні джерела є optional або залежать від capability конкретного MikroTik, це треба написати прямо.

---

### 4. Дописати розділ про state model і time fields

Потрібно окремо і коротко описати:

- `online`
- `idle`
- `offline`
- `unknown`

І пов'язані поля:

- `state_changed_at`
- `online_since`
- `idle_since`
- `offline_since`

Також треба пояснити:
- коли ці поля заповнюються
- коли вони можуть бути `null`
- як `unknown` поводиться щодо time fields
- як це впливає на default sorting та UI behavior

---

### 5. Дописати snapshot schema та event schema

Потрібно додати короткий, практичний опис:

- які основні поля snapshot device object вважаються стабільними
- які поля derived / calculated
- які поля можуть бути порожні
- які події записуються у `events.jsonl`
- мінімальні приклади:
  - один snapshot device object
  - один `FIELD_CHANGE`
  - один `state_changed`
  - один `device_offline` або аналогічний event

---

### 6. Дописати troubleshooting для expected warnings

Потрібно окремо додати кейси, які не є реальною помилкою, але можуть виглядати як проблема в логах.

Наприклад:
- `/interface/wireless` -> `no such command prefix`
- відсутність wireless support на конкретному пристрої
- expected capability mismatch
- ситуації, коли collector продовжує роботу без частини optional data

Мета:
- прибрати двозначність
- відокремити expected conditions від real errors

---

### 7. Додати operator-oriented verification section

Потрібно додати короткий практичний розділ:
- як перевірити, що snapshots створюються
- як перевірити, що `events.jsonl` пишеться
- як перевірити останні diff errors
- як перевірити volume mapping
- як швидко зрозуміти, чи persistence працює коректно

Це має бути зручно для реальної експлуатації.

## Критерії приймання

Задача вважається виконаною, якщо:

1. README узгоджений з поточним кодом і runtime behavior
2. deployment examples більше не суперечать один одному

7. додано короткий operator-oriented verification section
