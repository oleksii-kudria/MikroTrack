# TASK-037 - Replace Flags with semantic badges (PERM / STATIC / DYNAMIC)

## Контекст

Поточний UI відображає технічні прапорці (Flags / state), які важко читати:
- DC, SC, D, тощо
- користувач повинен розуміти внутрішню логіку MikroTik

Це створює зайве когнітивне навантаження.

## Мета

Замінити технічні flags на прості та зрозумілі semantic badges:

- PERM
- STATIC
- DYNAMIC

Без відображення технічних джерел (DHCP / ARP).

## Важливо

- Backend НЕ змінюється
- Всі дані вже є, змінюється тільки UI layer
- Поточна логіка збору та агрегації НЕ ламається

---

## Вимоги

### 1. Прибрати колонку

Видалити колонку:

```
Flags / state
```

---

### 2. Додати нову колонку / або замінити

Використати цю ж позицію для відображення badge.

---

### 3. Логіка визначення badge

Пріоритет:

```
PERM > STATIC > DYNAMIC
```

#### Правила:

```
IF arp_is_static == true:
    badge = "PERM"

ELIF dhcp_is_static == true:
    badge = "STATIC"

ELSE:
    badge = "DYNAMIC"
```

---

### 4. НЕ використовувати

Заборонено відображати:

- DHCP
- ARP
- bridge_host
- будь-які технічні flags (DC, SC, D)

---

### 5. Відображення

#### Приклади:

```
[PERM]
[STATIC]
[DYNAMIC]
```

---

### 6. Interface MAC

Для:

```
entity_type == "interface"
```

Залишаємо існуючу логіку:

```
[INTERFACE] ether2
```

Badge PERM/STATIC/DYNAMIC НЕ застосовується.

---

### 7. Стилі (рекомендація)

| Badge   | Колір |
|--------|------|
| PERM   | фіолетовий / зелений |
| STATIC | синій |
| DYNAMIC| сірий / світло-блакитний |

---

### 8. Acceptance Criteria

- колонка Flags повністю відсутня
- для кожного пристрою є один badge (PERM/STATIC/DYNAMIC)
- badge визначається коректно за пріоритетом
- interface записи не отримують ці badge
- UI стає простішим і зрозумілішим

---

## Очікуваний результат

UI відображає не технічні прапорці, а сенс:

- PERM → закріплений пристрій
- STATIC → зарезервований
- DYNAMIC → звичайний клієнт

Користувач може швидко оцінити стан мережі без розуміння внутрішніх деталей RouterOS.
