# TASK-099 - Add mac_vendor to all devices using local mac_vendors.json

## Опис задачі

Необхідно інтегрувати локальну базу MAC-вендорів у pipeline побудови пристроїв, щоб кожен пристрій у unified device model отримував поле:

`mac_vendor`

Джерело даних для визначення вендора:
- тільки локальний файл `app/data/mac_vendors.json`
- без зовнішніх API під час runtime

Ціль задачі:
- під час обробки пристроїв визначати вендора за MAC-адресою
- додавати `mac_vendor` у модель пристрою, snapshots та API-відповідь
- зберегти offline-first підхід

---

## Що необхідно реалізувати

### 1. Інтеграція у pipeline обробки пристроїв

Необхідно інтегрувати lookup вендора у поточний pipeline побудови unified device model.

Для кожного пристрою:
- взяти `mac`
- визначити OUI
- знайти vendor у локальній базі
- записати результат у `mac_vendor`

### Правила:
- якщо MAC відсутній або невалідний → `mac_vendor = null`
- якщо OUI не знайдено → `mac_vendor = null`
- якщо MAC random / locally administered → vendor не вважається надійним, але поле `mac_vendor` можна заповнювати як informational only, якщо lookup спрацював

---

### 2. Поле mac_vendor у device model

Необхідно додати поле:

`mac_vendor: string | null`

у всі місця, де формується або серіалізується unified device model.

### Необхідно перевірити:
- builder / normalize / merge logic
- snapshot serialization
- API schema
- UI-ready response fields

---

### 3. Використання локальної бази

Runtime повинен використовувати тільки локально завантажену базу MAC-вендорів.

### Вимоги:
- не виконувати HTTP-запити
- не звертатися до IEEE під час runtime
- використовувати вже реалізований loader для `mac_vendors.json`

---

### 4. Поведінка для random MAC

Необхідно врахувати існуюче поле:

`is_random_mac`

### Поведінка:
- random MAC не повинен вважатися надійною основою для ідентифікації вендора
- якщо vendor знайдений, його можна повертати як informational value
- логіка не повинна зупиняти обробку пристрою

---

### 5. Snapshot та API

Необхідно переконатися, що `mac_vendor`:
- зберігається у snapshot
- повертається у `/api/devices`
- стабільно серіалізується у JSON
- не ламає існуючу сумісність

---

### 6. Backward compatibility

Необхідно забезпечити:
- коректну роботу для старих snapshot, де поля `mac_vendor` ще немає
- відсутність падіння при читанні історичних даних
- стабільну поведінку API для mixed data

## Тести

Необхідно додати або оновити тести для таких сценаріїв:

1. відомий MAC → `mac_vendor` заповнюється
2. невідомий MAC → `mac_vendor = null`
3. невалідний MAC → `mac_vendor = null`
4. random MAC → обробка не падає
5. `mac_vendor` потрапляє у snapshot
6. `mac_vendor` повертається через API
7. старі snapshot без `mac_vendor` читаються коректно

---

## Acceptance Criteria

Задача вважається виконаною якщо:

1. Кожен пристрій у unified device model має поле `mac_vendor`
2. Значення береться з локального `app/data/mac_vendors.json`
3. Runtime не використовує зовнішні API
4. Snapshot містять `mac_vendor`
5. `/api/devices` повертає `mac_vendor`
6. Для невідомих або невалідних MAC використовується `null`
7. Старі snapshot без цього поля не ламають систему
8. Додані або оновлені тести
9. Оновлена документація українською та англійською
10. Логи залишаються англомовними та контрольованими
