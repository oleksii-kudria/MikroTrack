# TASK-095 - Add MAC metadata fields: random flag and vendor detection

## Опис

Необхідно розширити модель пристрою новими полями, пов’язаними з MAC-адресою.

Потрібно додати:

1. поле, яке явно показує, чи є MAC-адреса рандомною
2. поле, яке містить назву вендора, якщо її можна визначити за MAC

Важливо:
- зміни потрібні тільки на backend / data model / API level
- у Web UI нічого змінювати не треба
- потрібно розробити сам механізм визначення цих даних

---

## Що потрібно зробити

### 1. Додати нові поля в модель пристрою

Для кожного device object додати:

- `is_random_mac` - `true` / `false`
- `mac_vendor` - назва вендора або `null` / порожнє значення, якщо визначити не вдалося

Очікуваний приклад:

```json
{
  "mac": "9A:CD:38:5D:2E:0A",
  "is_random_mac": true,
  "mac_vendor": null
}
```

або:

```json
{
  "mac": "20:37:A5:87:2A:13",
  "is_random_mac": false,
  "mac_vendor": "Apple, Inc."
}
```

---

### 2. Реалізувати визначення random MAC

Потрібно додати окремий механізм визначення, чи є MAC рандомною.

Очікувана логіка:
- визначення має базуватись на бітовій структурі MAC address
- результат має бути deterministic
- значення `badges: ["RANDOM"]` не повинно бути єдиним джерелом істини
- `is_random_mac` має обчислюватися напряму з MAC-адреси

Важливо:
- існуючий badge `RANDOM` можна зберегти як UI/helper behavior
- але нове поле має бути окремим нормалізованим backend field

---

### 3. Реалізувати визначення вендора по MAC

Потрібно розробити механізм визначення вендора по MAC address.

Мінімальні вимоги:

- визначення по OUI / префіксу MAC
- якщо вендор знайдено - записати його у `mac_vendor`
- якщо вендор не знайдено або MAC random/local - повертати `null` або інше чітко визначене порожнє значення

Важливо:
- якщо MAC random/local, `mac_vendor` не повинен виглядати як достовірний vendor result
- треба уникати хибної впевненості

---

### 4. Вибрати і задокументувати джерело vendor mapping

Потрібно визначити, як саме проект отримує vendor mapping:

Можливі варіанти:
- локальний OUI файл у репозиторії
- окремий data file, який оновлюється вручну
- lightweight embedded vendor map
- інший offline-friendly механізм

Перевага:
- offline / deterministic behavior
- без залежності від зовнішнього API під час runtime

Потрібно задокументувати:
- де лежить vendor data
- як вона оновлюється
- що відбувається, якщо OUI не знайдено

---

### 5. Інтегрувати нові поля в unified device model

Нові поля мають бути доступні у фінальному device object, який:
- зберігається в snapshot

- використовується backend logic

Тобто:
- поля мають бути частиною стабільної схеми
- не helper-only локальною змінною

---

- нові snapshot-и вже містили нові поля
- API не падав на старих даних

- нові поля `is_random_mac` і `mac_vendor`

- які обмеження є у vendor detection
- де знаходиться vendor mapping data

## Тести

Додати мінімальні тести для:

### Random MAC
- globally unique MAC -> `is_random_mac=false`
- locally administered / random MAC -> `is_random_mac=true`

### Vendor lookup
- відомий OUI -> коректний vendor
- невідомий OUI -> `null`
- random MAC -> `is_random_mac=true`, vendor handled predictably

### Model integration
- нові поля присутні у фінальному device object
- snapshot serialization працює
- API output містить нові поля

## Критерії приймання

Задача вважається виконаною, якщо:

1. додано поле `is_random_mac`
2. додано поле `mac_vendor`
3. random MAC визначається окремою backend logic, а не тільки через badge
4. vendor визначається по MAC/OUI без зовнішньої runtime залежності
5. нові поля присутні у snapshot і API output
6. Web UI не змінюється
7. додано тести
8. документація оновлена українською та англійською
