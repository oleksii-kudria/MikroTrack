# TASK-087 - Add operator playbook for runtime verification and incident diagnostics

## Опис

Перед переходом до Phase 2 потрібно підготувати короткий, практичний operator playbook для експлуатації MikroTrack.

Зараз основна логіка вже працює, але для реального використання не вистачає одного зручного документа, який допомагає швидко відповісти на питання:

- чи collector зараз працює
- чи snapshots створюються
- чи `events.jsonl` записується
- чи diff проходить без помилок
- чи API віддає актуальні дані
- чи Docker volume mapping налаштований правильно
- як швидко знайти останню реальну проблему

Ця задача не додає новий функціонал. Вона створює операційний playbook для швидкої перевірки стану системи та первинної діагностики проблем.

---

## Що потрібно зробити

### 1. Створити окремий operator playbook document

Потрібно додати окремий документ, наприклад:

- `docs/operator-playbook.md`

Документ має бути коротким, практичним і орієнтованим на реальну експлуатацію, а не на розробку.

---

### 2. Додати розділ "Quick health check"

Потрібно описати базову послідовність перевірок:

- контейнер `mikrotrack-app` запущений
- контейнер `mikrotrack-web` запущений
- collector виконує collection cycle
- snapshots створюються
- `events.jsonl` існує та оновлюється
- API відповідає
- Web UI відкривається

Приклади команд мають бути реальними і готовими до копіювання.

---

### 3. Додати перевірку persistence

Окремо описати:

- як перевірити вміст `PERSISTENCE_PATH`
- як перевірити, що snapshot-файли зростають по часу
- як перевірити наявність `events.jsonl`
- як перевірити права на директорію
- як переконатися, що volume mapping працює

---

### 4. Додати перевірку diff та events

Потрібно описати:

- як знайти останні `DIFF_ERROR`
- як подивитися `Diff summary`
- як перевірити `Events persisted`
- як знайти traceback
- як відрізнити, що проблема в diff, а не в collector

---

### 5. Додати перевірку API

Потрібно описати:

- як перевірити `/api/devices`
- як швидко побачити, що API повертає не порожній список
- як перевірити, що у відповідях є ключові поля:
  - `status`
  - `state_changed_at`
  - `online_since`
  - `idle_since`
  - `offline_since`

---

### 6. Додати розділ "Expected warnings vs real errors"

Окремо описати, що вважати нормальним:

- optional wireless interface collection is not supported on some devices
- capability mismatch warnings
- відсутність `events.jsonl` на першому snapshot або при відсутності змін

І що вже є реальною проблемою:

- `DIFF_ERROR`
- `PERSISTENCE_ERROR`
- repeated API connection failures
- traceback в логах

---

### 7. Додати мінімальний incident response flow

Описати просту послідовність дій:

1. перевірити, чи контейнер працює
2. перевірити, чи йде collection cycle
3. перевірити snapshots
4. перевірити diff/events
5. перевірити API
6. перевірити volume mapping
7. зібрати логи для аналізу

---

### 8. Додати готові команди для діагностики

Потрібно включити practical commands на кшталт:

```bash
docker ps
docker logs mikrotrack-app --tail 200
docker exec -it mikrotrack-app sh -c 'ls -lah /data/snapshots'
curl http://127.0.0.1:8000/api/devices
```

Команди повинні бути короткі, корисні і без зайвого шуму.
