# TASK-042 - Add sorting for all table columns with state-aware ordering

## Контекст

У таблиці Devices відображаються:
- MAC
- IP
- Hostname
- Comments
- Assignment
- Session / state time

Наразі сортування або відсутнє, або недостатньо гнучке.

Потрібно додати можливість сортування по кожному полю та окремо врахувати спеціальну логіку сортування по статусу пристрою.

---

## Мета

1. Додати сортування по кожному стовпчику таблиці
2. Реалізувати сортування по статусу з фіксованим пріоритетом
3. Для пристроїв з однаковим статусом враховувати час поточного стану
4. Додати підтримку сортування в обох напрямках

---

## Вимоги

### 1. Загальне сортування

Додати можливість сортування по всіх полях:

- MAC
- IP
- Hostname
- Comments
- Assignment
- Session / state time
- Status

---

### 2. Сортування по статусу

Ввести окрему логіку сортування по статусу.

#### ASC (прямий порядок)

```text
online
idle
unknown
offline
```

#### DESC (зворотний порядок)

```text
offline
unknown
idle
online
```

---

### 3. Сортування всередині одного статусу

Якщо декілька пристроїв мають однаковий статус, додатково сортувати їх за тривалістю поточного стану.

#### Для ASC

Новіші зверху:

- найменший таймер зверху
- найбільший таймер знизу

Приклад для `online`:

```text
00:02
00:10
01:15
03:20
```

#### Для DESC

Старіші зверху:

```text
03:20
01:15
00:10
00:02
```

---

### 4. Який час використовувати

Для сортування за статусом використовувати відповідний timestamp поточного стану:

- `online` → online_since
- `idle` → idle_since
- `offline` → offline_since
- `unknown` → state_changed_at (або fallback logic, якщо окремого поля немає)

Тобто сортування повинно базуватись не на відформатованому рядку таймера, а на raw timestamps.

---

### 5. Сортування стовпчика Session / state time

При сортуванні по стовпчику `Session / state time` використовувати той самий принцип:

- порівнювати raw timestamps
- не сортувати по formatted string (`17:05:12`, `43:31`, `-`)

---

### 6. Сортування текстових полів

Для:
- MAC
- IP
- Hostname
- Comments
- Assignment

реалізувати стандартне сортування:

- ASC
- DESC

#### Додатково для IP
IP потрібно сортувати як IP-адресу, а не як текст.

Приклад:
- 192.168.36.9
- 192.168.36.10
- 192.168.36.100

мають сортуватись коректно.

---

### 7. Null / empty values

Порожні значення:

- `-`
- null
- empty string

повинні стабільно сортуватись внизу для ASC і вгорі для DESC
або зафіксовано одним чітким правилом для всіх стовпчиків.

Обрати та зафіксувати одну поведінку.

---

### 8. UI requirements

- клік по заголовку стовпчика перемикає сортування:
  - none → ASC → DESC
- показувати індикатор сортування у заголовку
- активне сортування повинно бути зрозумілим для користувача

---

### 9. Acceptance Criteria

- можна сортувати по кожному стовпчику
- статус сортується за спеціальним пріоритетом:
  - online → idle → unknown → offline
- у межах одного статусу новіші пристрої зверху для ASC
- у межах одного статусу старіші зверху для DESC
- IP сортуються коректно як IP
- formatted timers не використовуються для порівняння
- UI показує активний напрямок сортування

---

## Очікуваний результат

Користувач може швидко:
- вивести найновіші online пристрої нагору
- побачити offline записи окремо
- сортувати таблицю по будь-якому полю
- працювати з таблицею як з повноцінним операційним інструментом
